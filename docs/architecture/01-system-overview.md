# System Overview

JewelloSoft is designed as an "offline-first" desktop application tailored for jewelry retail. Unlike traditional web applications that rely entirely on a remote server, JewelloSoft bundles its own web server and database directly onto the user's machine. 

This ensures that shop operations, specifically billing and inventory management, never halt due to internet connectivity issues.

## Tech Stack

The application is split into three main logical components:

### 1. The Backend (Django)
- **Framework**: Django & Django REST Framework (DRF).
- **Database**: SQLite (local flat-file database stored in the backend directory).
- **Web Server**: Waitress (used instead of standard WSGI/Gunicorn because Waitress runs natively and reliably on Windows without Cygwin/WSL).
- **Compilation**: The entire Python backend is compiled into a single executable (`backend.exe`) using **PyInstaller**. This prevents the end-user from needing Python installed on their machine.

### 2. The Frontend (React)
- **Framework**: React.js built with Vite.
- **Styling**: Vanilla CSS with a custom "Noir Gold" design system.
- **Role**: Provides the user interface, makes REST API calls to the local Django server, and generates standard HTML/CSS templates that are converted to A4 PDFs for printing.

### 3. The Desktop Wrapper (Electron)
- **Framework**: Electron.
- **Role**: Acts as the bridge. When the user double-clicks the JewelloSoft icon, Electron launches. During its initialization (`main.js`), it spawns a hidden background process that runs `backend.exe`. Once the backend confirms it is listening on a local port, Electron loads the bundled React frontend into its Chromium window.

## How They Communicate

1. **Launch**: Electron starts -> Spawns `backend.exe` -> Waits for localhost port to be active.
2. **UI Load**: Electron loads the compiled React static files (`index.html`).
3. **Data Fetch**: React makes standard `fetch()` or `axios` HTTP requests to `http://localhost:<PORT>/api/`.
4. **Auth**: JWT (JSON Web Tokens) are generated by the backend and stored in the frontend's local storage to authenticate all subsequent API calls.
5. **Shutdown**: When the user closes the Electron window, the main process catches the exit event and sends a termination signal to `backend.exe` to prevent orphan processes.
